# Featherless filter tag values (harvested, not API-derived)

These are the actual filter option values from the featherless.ai/models filter UI, hand-harvested by
Alex on 2026-07-01 since the public API doesn't expose them anywhere (see docs/providers/featherless-notes.md —
no facets endpoint, no single-model detail endpoint, values aren't echoed in the `/v1/models` list
response even though the API silently accepts them as query filters).

**These are not an official, defined taxonomy.** Confirmed against HuggingFace's own docs: `capabilities`,
`domains`, `creative`, and `training` are almost certainly HuggingFace's free-form, self-reported `tags`
model card field (arbitrary strings, no validation, "you can add any tag you want" per HF's docs) —
not their curated `pipeline_tag`. `families`, `architectures`, `parameter_bucket`, and `modalities` look
structurally cleaner and are probably computed by Featherless from actual model config rather than
scraped from free text, but that's inference, not confirmed.

**Treat all of these as soft/unverified signal, especially `creative`.** A model tagged `erp` was
labeled that way by whoever uploaded it, not tested by anyone. This is exactly why loremaster.md's
Future Phases section says to surface provider/model content-permissiveness as a visible flag rather
than hard-filter on it.

Full values, plus a filter/never/neutral verdict and 0-5 usefulness ratings per agent role (worker/
editor/author), live in `src/data/featherless-tag-ratings.json` (generated by
`scripts/generate-tag-ratings.ts` — edit the overrides there and regenerate, don't hand-edit the JSON).
That file is wired into `src/inference/featherless-tag-ratings.ts` (query helpers) and
`src/inference/featherless-models.ts`'s `suggestFiltersForRole(role)`, which builds a starting
`listModels` filter for a given role. Verified working end-to-end 2026-07-01
(`scripts/test-role-suggestions.ts`) — author-role queries surfaced real community RP/ERP fine-tunes
(Pantheon-RP, Heretic-Uncensored, wifuGPT, etc.), worker/editor queries stayed tool-calling/instruct-focused.

Only 7 of 453 tags are hard `always`/`never` (deliberately conservative — these are self-reported,
unverified community tags, so soft quality/content judgments stay `neutral` with an adjusted star
rating rather than a hard filter). `always`: `modalities=text`. `never`: `modalities=embedding`,
`capabilities=image-generation`/`video-generation`, `families=bert`/`nomic-bert`/`xlm-roberta` — all
structurally non-generative or non-text, not soft judgment calls.

**Important limitation:** this only helps _search_ for candidate models — the `/v1/models` list response
doesn't echo back which tags a given model actually has, so there's no way to score an already-fetched
model against these ratings after the fact, only to bias the query itself.

## Values most relevant to Loremaster

- `modalities=text` — the only modality we want (`vision`, `embedding` are out of scope; see docs/providers/featherless-notes.md for what `embedding` even means).
- `capabilities=tool-use` — required for Worker (and Editor, once built) roles that need forced tool-calling. Confirmed to correlate imperfectly with real tool-calling ability — verify empirically, per docs/providers/featherless-notes.md's Heretic-model finding.
- `creative=roleplay`, `creative=erp` — candidate signals for Author-role model discovery. Semantics of combining multiple `creative` values (AND vs OR) not yet tested — see below.
- `training=abliterated`, `training=uncensored` — signals for permissive/uncensored fine-tunes, relevant to Author-role selection given loremaster.md's model-selection-as-guardrail philosophy. Also self-reported, same caveat applies.

## Combining multiple values within one filter dimension = OR

Confirmed 2026-07-01: both `creative=erp,roleplay` (comma) and `creative=erp&creative=roleplay`
(repeated param) return the **union** of the two, not the intersection. Test: `creative=erp` alone
totaled 51, `creative=roleplay` alone totaled 564, combined (either syntax) totaled 567 — matching the
client-computed union (304 unique IDs fetched) and clearly not the intersection (25 unique IDs). So
"erp OR roleplay" can be expressed directly as a single server-side query; no need to fetch separately
and merge client-side for this case.

Unrelated quirk noticed during this test, not yet chased down: `per_page=500` didn't actually return
500 results even when `total` was well above that — pagination appears capped below the requested
`per_page` somewhere. Not urgent since our real queries won't need thousands of results at once, but
don't assume `per_page` is honored at face value if a future feature depends on exact result counts.
